# 四角色架构规范

> **用途**：Agent体系审查的核心方法论，定义四类角色的职责边界和禁止事项
> **版本**：v1.0 | 2026-06-17
> **来源**：[[Wiki/sources/agent审查/01-agent-architecture-guide]]

## 一、架构总览

```
用户请求
    │
    ▼
┌─────────────────────┐
│   主 agent           │  ← 唯一面向用户的入口
│   (Orchestrator)    │
└──────────┬──────────┘
           │ 分发子任务
     ┌─────┼─────┐
     ▼     ▼     ▼
  子 agent  子 agent  子 agent   ← 专项执行，不直接响应用户
     └─────┬─────┘
           │ 汇聚结果
           ▼
┌─────────────────────┐
│   报告生成 agent     │  ← 结构化输出
└──────────┬──────────┘
           │
           ▼
┌─────────────────────┐
│   审查 agent         │  ← 质量把关，可触发返工
└──────────┬──────────┘
           │ 通过
           ▼
        最终交付
```

## 二、四角色职责边界

### 2.1 主 Agent（Orchestrator）

**核心职责**：
```
┌──────────────────────────────────────────────────────────────┐
│                    主Agent职责边界                            │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  【必须做的事】                                               │
│  ├─ 理解用户意图，将任务拆解为独立子任务                       │
│  ├─ 维护子Agent注册表，按能力路由任务                         │
│  ├─ 协调并行/串行执行顺序                                    │
│  ├─ 聚合子Agent结果，判断是否满足交付条件                     │
│  ├─ KVP验证：确认子Agent被调用且有有效输出                    │
│  └─ 错误处理：子Agent调用失败时的重试机制                     │
│                                                              │
│  【不该做的事】                                               │
│  ├─ 不直接执行具体业务逻辑（搜索、计算、写文件等）             │
│  ├─ 不直接生成最终报告                                       │
│  └─ 不做质量审查（委托审查Agent）                            │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

**调度消息格式（发出）**：
```json
{
  "task_id": "t-001",
  "task_type": "collect|analyze|transform",
  "input": { ... },
  "output_schema": { ... },
  "timeout_s": 30,
  "priority": "high|normal|low"
}
```

**聚合消息格式（接收）**：
```json
{
  "task_id": "t-001",
  "status": "success|failed|partial",
  "output": { ... },
  "error": null
}
```

---

### 2.2 子 Agent（Specialist）

**核心职责**：
```
┌──────────────────────────────────────────────────────────────┐
│                    子Agent职责边界                            │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  【必须做的事】                                               │
│  ├─ 单一领域的深度执行（每个子Agent只做一件事）               │
│  ├─ 严格按照输入契约接收任务                                 │
│  ├─ 严格按照输出契约返回结果                                 │
│  └─ 失败时上报错误，不自行决策重试策略                       │
│                                                              │
│  【禁止事项】                                                 │
│  ├─ 不得直接响应用户输入                                     │
│  ├─ 不得调用其他子Agent（防止环路）                          │
│  ├─ 不得自行修改任务范围                                     │
│  └─ 不得自行决策重试策略                                     │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

**子Agent类型示例**：

| 类型 | 职责 | 典型工具 |
|------|------|---------|
| 数据采集Agent | 从外部源获取原始数据 | web_search、API调用 |
| 分析Agent | 对数据执行计算/推理 | code_execution、数学工具 |
| 转换Agent | 格式转换、数据清洗 | 文件操作、正则处理 |
| 写作Agent | 生成特定风格文本 | LLM生成 |

---

### 2.3 报告生成 Agent（Reporter）

**核心职责**：
```
┌──────────────────────────────────────────────────────────────┐
│                    输出Agent职责边界                          │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  【必须做的事】                                               │
│  ├─ 接收主Agent聚合的结构化数据                               │
│  ├─ 按固定模板生成可读报告                                   │
│  ├─ 输出格式确定，不依赖上下文判断                           │
│  └─ 标注数据来源（每个章节来自哪个Agent）                     │
│                                                              │
│  【触发条件】                                                 │
│  └─ 仅当主Agent确认所有子任务完成、数据聚合完毕后触发        │
│                                                              │
│  【输出类型】                                                 │
│  └─ Markdown / PDF / DOCX / JSON，由调用方指定              │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

---

### 2.4 审查 Agent（Reviewer）

**核心职责**：
```
┌──────────────────────────────────────────────────────────────┐
│                    审查Agent职责边界                          │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  【必须做的事】                                               │
│  ├─ 对报告生成Agent的输出进行质量审核                        │
│  ├─ 返回结构化审查结果（通过/不通过+具体问题）                │
│  └─ 不自行修改报告内容，只出具意见                           │
│                                                              │
│  【独立性】                                                   │
│  ├─ 不参与生产流程                                           │
│  └─ 只在生产完成后介入，保证审查客观性                        │
│                                                              │
│  【升级条件】                                                 │
│  └─ score < 60 或存在任意 severity: error 的问题时，         │
│     向主Agent返回 require_revision: true，触发返工流程       │
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

**审查结果格式**：
```json
{
  "passed": true,
  "score": 85,
  "issues": [
    {
      "dimension": "completeness",
      "severity": "warning|error",
      "location": "第三节",
      "description": "缺少数据来源说明"
    }
  ],
  "suggestions": ["在第三节末尾补充数据来源链接"],
  "require_revision": false
}
```

## 三、消息协议规范

### 3.1 通用消息头

所有Agent间通信均携带以下字段：

```json
{
  "msg_id": "uuid-v4",
  "from": "orchestrator|sub-agent-{name}|reporter|reviewer",
  "to": "orchestrator|sub-agent-{name}|reporter|reviewer",
  "timestamp": "ISO-8601",
  "conversation_id": "会话级唯一ID",
  "payload": { ... }
}
```

### 3.2 状态流转

```
pending → running → success
                 → failed（可重试）
                 → aborted（不可重试）
```

重试策略由主Agent统一决策，子Agent只上报失败，不自行重试。

## 五、错误处理规范

| 错误类型 | 处理方 | 动作 |
|---------|--------|------|
| 子任务超时 | 主Agent | 重试一次，仍失败则标记partial并继续 |
| 子任务数据缺失 | 主Agent | 评估是否影响最终交付，决定是否终止 |
| 报告格式不合规 | 审查Agent | 返回require_revision，主Agent触发返工 |
| 审查未通过超过2次 | 主Agent | 升级人工介入，停止自动流程 |
| 消息格式错误 | 接收方 | 返回400错误，记录日志，不处理 |

## 六、六大设计原则

```
┌──────────────────────────────────────────────────────────────┐
│                    六大设计原则                                │
├──────────────────────────────────────────────────────────────┤
│                                                              │
│  【原则一：单一职责】                                         │
│  每个Agent只做一件事。判断标准：能否用一个动词描述其全部工作？ │
│                                                              │
│  【原则二：契约优先】                                         │
│  输入/输出schema先于实现定义，Agent之间通过契约解耦。         │
│                                                              │
│  【原则三：主Agent集权】                                      │
│  所有决策权（重试、终止、路由）归主Agent，子Agent不做策略判断。│
│                                                              │
│  【原则四：审查独立】                                         │
│  审查Agent不参与生产流程，只在生产完成后介入，保证审查客观性。 │
│                                                              │
│  【原则五：失败可观测】                                       │
│  每次失败都必须有结构化日志，包含msg_id、from、error_code。   │
│                                                              │
│  【原则六：KVP验证】                                          │
│  主Agent必须验证子Agent输出，禁止在未验证时标记"called":true。│
│                                                              │
└──────────────────────────────────────────────────────────────┘
```

## 七、角色对照表

| 角色 | 核心能力 | 决策权 | 是否面向用户 |
|------|----------|--------|-------------|
| 主Agent | 任务拆解、路由、协调 | 有 | 是（唯一入口） |
| 子Agent | 单一领域执行 | 无 | 否 |
| 报告生成Agent | 结构化输出 | 无 | 否 |
| 审查Agent | 质量把关 | 无（仅出具意见） | 否 |
